# สรุป: จากเอกสารหนึ่งหน้าไปสู่ประวัติที่แบ่งปันได้

## เป้าหมายที่คุณทำสำเร็จ

คุณไม่ได้เพียงจำ tag หรือ property แต่เดินครบวงจรของเว็บชิ้นเล็ก:

```text
ปัญหาและผู้ใช้
  ↓
HTML บอกโครงสร้างและความหมาย
  ↓
CSS กำหนดการนำเสนอและปรับตามหน้าจอ
  ↓
ตรวจด้วย browser, keyboard และ validator
  ↓
Git บันทึก milestone
  ↓
GitHub รับสำเนาเพื่อแบ่งปัน
```

## Mental model ที่ควรติดตัว

- **Web ไม่ใช่ Internet:** Web เกิดเพื่อเชื่อมข้อมูลบนเครือข่าย และ HTML รุ่นแรกเป็นส่วนหนึ่งของระบบนั้น
- **HTML ถามว่า “นี่คืออะไร”:** heading, navigation, main content และ image alternative มีความหมายต่อทั้งคนและเครื่องมือช่วยเหลือ
- **CSS ถามว่า “ควรนำเสนออย่างไร”:** การแยก presentation ทำให้เปลี่ยนภาพลักษณ์และ layout ได้โดยไม่ทำลายโครงสร้าง
- **Responsive เป็นเงื่อนไข ไม่ใช่ขนาดเดียว:** ตรวจจอเล็ก กลาง ใหญ่ และเนื้อหาจริง
- **Git บันทึกการตัดสินใจ:** `status` บอกสถานะ, `diff` บอกสิ่งที่เปลี่ยน, `add` เลือก snapshot, `commit` อธิบาย milestone และ `push` ส่ง commit

## ตรวจตัวเองก่อนจบ

- [ ] อธิบายได้ว่า Web เกิดจากปัญหาใด และใครเขียน HTML รุ่นแรก
- [ ] เปิด source แล้วชี้ semantic regions และ alt decision ได้
- [ ] ใช้ Tab ครบหน้าและเห็น focus ทุกจุด
- [ ] หน้าไม่ล้นที่ 320, 768 และ 1280px
- [ ] อ่าน `git status`, `git diff` และ `git log --oneline` ได้
- [ ] แยกได้ว่า commit อยู่ในเครื่อง ส่วน push ส่งขึ้น remote

## เลือกก้าวถัดไปเพียงหนึ่งก้าว

อย่ารีบกระโดดไป framework ก่อนพื้นฐานแน่น เลือกหนึ่งทางในสัปดาห์หน้า:

1. **เนื้อหามากขึ้น:** ทำ [ตาราง](/html/lists-tables) และ [ฟอร์ม](/html/forms) โดยยังไม่ทำ backend
2. **Layout มากขึ้น:** ทำ [CSS Grid](/css/grid) และเปรียบเทียบกับ Flexbox
3. **Workflow มากขึ้น:** clone repository ลงโฟลเดอร์ใหม่ แก้หนึ่งเรื่อง แล้ว commit/push
4. **คุณภาพมากขึ้น:** ให้คนอื่นทดสอบ keyboard และจอเล็ก แล้วแก้จากหลักฐาน

เปิด [แหล่งเรียนรู้](/workshop/resources) เพื่อเลือกเอกสารที่ตรงกับทางนั้น และกลับมาใช้ rubric ของ [Capstone](/workshop/project) ตรวจงานชิ้นถัดไป

::: tip หลักการจบเวิร์กชอป
ชิ้นงานเล็กที่ตรวจได้และมีประวัติชัด มีค่ากว่าโปรเจกต์ใหญ่ที่อธิบายไม่ได้ว่าทำงานอย่างไร
:::
